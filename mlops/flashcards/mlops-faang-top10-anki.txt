What is training-serving skew, why is it dangerous, and how do you prevent it?	<b>What it is:</b> The features used during training differ from the features computed at serving time. The model was optimized on one distribution but receives another in production.<br><br><b>Why it's dangerous:</b> It's <i>silent</i> — no exception is thrown. The model deploys, offline metrics looked great, but online performance is worse and there's no obvious error.<br><br><b>Common causes:</b><ul><li>Training and serving teams duplicate feature logic independently, then diverge</li><li>Training reads from a batch warehouse; serving reads from a real-time stream with different null handling or aggregation windows</li><li>Preprocessing (scaler, encoder) is fit on training data but not serialized and reused at serving time</li><li>Data leakage: training features include future information (violates point-in-time correctness)</li></ul><b>Prevention:</b><ol><li>Use a <b>feature store</b> — single feature definition computed once, served to both training and online inference</li><li><b>Serialize preprocessing</b> — save the fitted scaler/encoder inside the model artifact (sklearn Pipeline, TF SavedModel with preprocessing layers)</li><li><b>Log serving features</b> — log actual feature values sent to the model; compare distributions against training data periodically</li><li><b>Shadow mode testing</b> — run the new model alongside the old one before go-live; compare feature distributions</li></ol>	MLOps
What is the difference between data drift, concept drift, and label drift?	All three are forms of <i>distribution shift</i> that cause a deployed model to degrade over time — without any code change.<br><br><table border="1" cellpadding="5" style="border-collapse:collapse"><tr><th>Type</th><th>What changes</th><th>Example</th></tr><tr><td><b>Data drift</b> (covariate shift)</td><td>Distribution of input features X</td><td>User age skews older as app grows</td></tr><tr><td><b>Concept drift</b></td><td>Relationship between X and target Y</td><td>Spam patterns change as spammers adapt to your classifier</td></tr><tr><td><b>Label drift</b></td><td>Distribution of the output Y</td><td>Fraud rate increases during a recession</td></tr><tr><td><b>Prediction drift</b></td><td>Distribution of model outputs</td><td>Model starts predicting one class disproportionately</td></tr></table><br><b>How to detect:</b><ul><li>Continuous features: Kolmogorov-Smirnov test or Population Stability Index (PSI)</li><li>Categorical features: Chi-squared test or PSI</li><li>PSI rule of thumb: &lt; 0.1 = fine, 0.1–0.25 = monitor, &gt; 0.25 = retrain</li></ul><b>How to respond:</b> alert → investigate (which features?) → trigger retraining → validate new model → promote	MLOps
Design a feature store. What are its components and why do you need one?	<b>Why you need it:</b> Without a feature store, teams recompute the same features independently in training code and serving code — causing training-serving skew and duplicated effort.<br><br><b>Core components:</b><br><div style="font-family:monospace;background:#f5f5f5;padding:8px;border-radius:4px;margin:4px 0">Raw data → Feature pipeline → Offline store &nbsp;(training, batch scoring)<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;→ Online store &nbsp;&nbsp;(real-time inference, low latency)</div><ul><li><b>Offline store:</b> historical feature values; backed by a data warehouse (BigQuery, Hive, S3 + Parquet). Used to build training datasets.</li><li><b>Online store:</b> latest feature values; backed by a low-latency KV store (Redis, DynamoDB, Bigtable). Queried at prediction time in milliseconds.</li><li><b>Feature pipelines:</b> compute features from raw data and write to both stores. Run on a schedule or event trigger.</li><li><b>Feature registry:</b> catalog of feature definitions, ownership, and lineage.</li></ul><b>Key design challenge — point-in-time correctness:</b> When building training datasets, you must use only features that were <i>available at the timestamp of each training example</i> — not features computed after the fact. Violating this causes data leakage.<br><br><b>Follow-up:</b> If a feature is not in the online store, either (1) add it to the materialization pipeline, or (2) compute it on-the-fly at serving time (acceptable only for cheap, fast transforms). Walk me through how you would deploy a new ML model to production with zero downtime.	<ol><li><b>Register the model:</b> After training, register the artifact to the model registry with metrics and lineage. Mark it <code>Staging</code>.</li><li><b>Offline validation:</b> Run automated checks — does the model beat the current production baseline on a held-out eval set? Does it pass latency/throughput requirements? Fail fast here.</li><li><b>Shadow mode deployment:</b> Deploy the new model alongside production but route its predictions to a log, not users. Compare prediction and feature distributions. Catch training-serving skew before any user impact.</li><li><b>Canary release:</b> Route a small slice of traffic (e.g., 1–5%) to the new model. Monitor online metrics (CTR, conversion, error rate) vs. the control. Gradually increase traffic if metrics hold.</li><li><b>Full rollout:</b> Shift 100% of traffic. Keep the old model warm for fast rollback.</li><li><b>Promote in registry:</b> Mark the new model <code>Production</code>; mark the old model <code>Archived</code>.</li></ol><b>Rollback:</b> Re-promote the previous version in the registry — it's already warm.	MLOps
What is continuous training (CT), and how does it differ from CI/CD?	<b>DevOps CI/CD:</b><ul><li><b>CI:</b> automatically build and test code on every commit</li><li><b>CD:</b> automatically deploy passing builds to production</li><li>Assumption: once deployed, software doesn't degrade on its own</li></ul><b>MLOps adds CT:</b><ul><li><b>CT (Continuous Training):</b> automatically retrain and redeploy the model when triggered</li><li>Necessary because models <i>do</i> degrade on their own — the world changes, data distributions shift, user behavior evolves — without any code change</li></ul><b>CT trigger strategies:</b><ul><li><b>Schedule:</b> retrain weekly/nightly on fresh data</li><li><b>Data volume:</b> trigger when N new labeled examples arrive</li><li><b>Drift threshold:</b> trigger when PSI or KS-stat exceeds a threshold</li><li><b>Performance degradation:</b> trigger when online metrics drop below a threshold</li></ul><b>CT pipeline steps:</b> data ingestion → validation → feature engineering → training → evaluation vs. baseline → conditional promotion<br><br><b>Key interview point:</b> CT is not "retrain whenever someone feels like it." It's an automated, gated pipeline with the same rigor as CI/CD.	MLOps
Describe the MLOps maturity levels. What does moving from Level 0 to Level 1 look like?	<i>Framework from Google's MLOps whitepaper.</i><br><br><table border="1" cellpadding="5" style="border-collapse:collapse"><tr><th>Level</th><th>Name</th><th>What it means</th></tr><tr><td><b>0</b></td><td>Manual</td><td>Data scientists run notebooks manually. Models are deployed by hand. No reproducibility, no automation.</td></tr><tr><td><b>1</b></td><td>Pipeline automation</td><td>Training is a versioned, automated pipeline. CT is enabled. Model registry exists.</td></tr><tr><td><b>2</b></td><td>CI/CD pipeline automation</td><td>The pipeline <i>itself</i> is built, tested, and deployed automatically on code changes. Full software engineering discipline applied to the ML system.</td></tr></table><br><b>Level 0 → Level 1 transition:</b><ol><li>Convert notebook code into modular, version-controlled Python steps</li><li>Wire steps into an orchestrated pipeline (Kubeflow, Vertex AI Pipelines, Metaflow, Airflow)</li><li>Add data validation at the pipeline entry point</li><li>Add experiment tracking (MLflow, W&amp;B) to every training run</li><li>Add a model registry with staging/production lifecycle</li><li>Build a CT trigger (schedule or drift-based)</li><li>Add automated evaluation gating before promotion</li></ol><b>Most companies are at Level 0 or early Level 1.</b> Start with data validation and experiment tracking — they give immediate value and are low-risk to add. How do you monitor a production ML model? What metrics do you track?	Monitoring spans four layers:<br><br><b>1. Infrastructure metrics</b> (same as any service)<ul><li>Latency (p50, p95, p99), throughput (QPS), error rate, CPU/GPU utilization</li></ul><b>2. Input data health</b><ul><li>Feature distribution drift (PSI, KS-test per feature)</li><li>Missing value rates, schema violations</li><li>Volume anomalies (sudden drop in traffic = data pipeline issue)</li></ul><b>3. Prediction health</b><ul><li>Prediction distribution drift (are outputs skewing to one class?)</li><li>Confidence score distribution</li></ul><b>4. Model performance</b> (requires ground truth labels — often delayed)<ul><li>Accuracy, AUC, precision/recall, RMSE depending on task</li><li>Track over rolling windows; alert on statistically significant degradation</li></ul><b>The challenge:</b> Ground truth is often delayed (days, weeks, or never). Input/prediction monitoring is a leading indicator you can act on <i>before</i> you have labels.<br><br><b>Alerting strategy:</b> set thresholds at two levels — <i>warn</i> (investigate) and <i>critical</i> (trigger CT pipeline or page on-call). How do you ensure reproducibility of a machine learning experiment?	Reproducibility means: given the same inputs, you can re-run a training job and get the same (or statistically equivalent) model. Required for debugging, auditing, and regulatory compliance.<br><br><b>Reproducibility checklist:</b><br><table border="1" cellpadding="5" style="border-collapse:collapse"><tr><th>What</th><th>How</th></tr><tr><td><b>Code</b></td><td>Log git commit SHA with every run</td></tr><tr><td><b>Data</b></td><td>Version your dataset (DVC, Delta Lake, LakeFS); log dataset hash or version ID</td></tr><tr><td><b>Environment</b></td><td>Pin library versions (requirements.txt, poetry.lock, or Docker image digest)</td></tr><tr><td><b>Random seeds</b></td><td>Set and log seeds for random, numpy, torch/tensorflow, data shuffle</td></tr><tr><td><b>Hyperparameters</b></td><td>Log all params via experiment tracker; never hardcode</td></tr><tr><td><b>Hardware</b></td><td>Note GPU type — some ops are non-deterministic across hardware</td></tr></table><br><b>Practical answer for interviews:</b> Use an experiment tracker (MLflow, W&amp;B) that logs all of the above automatically on every run, and use the model registry to link deployed models back to their exact training run.<br><br><b>Gotcha:</b> Full bit-for-bit reproducibility is often impossible with GPU training (non-deterministic CUDA ops). "Statistically equivalent" (same metrics within noise) is the practical goal.	MLOps
How would you version ML models and safely roll back a bad deployment?	<b>Versioning via a model registry:</b> Every trained model is registered as an immutable artifact with version number, metrics, lineage (dataset version, git SHA, experiment run ID), and lifecycle stage: <code>Staging</code> → <code>Production</code> → <code>Archived</code>.<br><br><b>Promotion flow:</b><div style="font-family:monospace;background:#f5f5f5;padding:8px;border-radius:4px;margin:4px 0">Training pipeline → registers v42 as Staging<br>Validation gate &nbsp; → passes eval, promotes v42 to Production (v41 → Archived)<br>Serving layer &nbsp;&nbsp;&nbsp; → always loads the model tagged Production</div><b>Rollback procedure:</b><ol><li>Detect degradation via monitoring alerts</li><li>In the model registry, re-promote the previous good version (e.g., v41) to <code>Production</code></li><li>Serving layer picks up the new tag on next refresh</li><li>v42 is demoted to <code>Archived</code> with a note</li><li>Post-mortem: what caused v42 to degrade? Data issue? Training bug?</li></ol><b>Key design principle:</b> The serving layer should be <i>tag-driven</i>, not version-number-driven. It loads "whatever is tagged Production" — so rollback is just a registry state change, not a redeployment.	MLOps
Design an end-to-end ML pipeline for a recommendation system at FAANG scale.	<i>Structure your answer in layers.</i><br><br><b>Data layer</b><ul><li>Raw events (clicks, views, purchases) land in a data lake (S3/GCS)</li><li>Batch pipeline (Spark/Beam) computes offline features → feature store offline store</li><li>Streaming pipeline (Kafka + Flink) computes real-time features → feature store online store (Redis)</li></ul><b>Training pipeline</b><ul><li>Trigger: daily schedule or drift detection</li><li>Steps: pull training data from feature store (point-in-time correct) → validate → train candidate generation model + ranking model → evaluate vs. baseline → register to model registry if better</li><li>Experiment tracking: every run logged with params, metrics, dataset version</li></ul><b>Serving layer</b><ul><li>Candidate generation: retrieve top-K candidates per user (ANN search, e.g., FAISS)</li><li>Ranking: score candidates with ranking model (online store feature lookup → model inference)</li><li>Latency budget: typically &lt;100ms total; online feature lookup must be &lt;10ms</li></ul><b>Deployment strategy</b><ul><li>New model enters shadow mode → canary (1%) → gradual rollout</li><li>Serving layer is tag-driven off model registry</li></ul><b>Monitoring</b><ul><li>Feature drift per feature group</li><li>CTR, conversion rate as online business metrics</li><li>Latency p99, error rate as infra metrics</li><li>Trigger CT pipeline if drift threshold exceeded</li></ul>	MLOps
