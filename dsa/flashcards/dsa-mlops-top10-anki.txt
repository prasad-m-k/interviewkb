LRU Cache — Design a data structure with O(1) <code>get(key)</code> and <code>put(key, value)</code> using LRU eviction. What two data structures do you combine and why?	<b>Data structures:</b> HashMap + Doubly Linked List (DLL)<br><br><b>Why the combination?</b><ul><li>HashMap → O(1) key lookup to find the node</li><li>DLL → O(1) move-to-front and remove-from-back <i>given a pointer</i> to the node</li><li>Neither alone suffices: HashMap can't track order; DLL is O(n) for lookup</li></ul><b>Design:</b><ul><li>DLL head = LRU end, tail = MRU end; two dummy sentinel nodes eliminate all edge cases</li><li><code>get(key)</code>: map lookup → remove node → insert at tail → return val</li><li><code>put(key, val)</code>: remove old node if exists; create node, insert at tail, add to map; if over capacity, remove <code>head.next</code> (LRU), delete from map</li></ul><b>Complexity:</b> O(1) get and put | O(capacity) space<br><br><b>Key gotcha:</b> dummy sentinel nodes are not optional style — they make every pointer operation uniform regardless of list size. Course Schedule II — Given numCourses and [course, prereq] pairs, return a valid ordering or [] if impossible. What algorithm detects cycles automatically?	<b>Algorithm:</b> Kahn's topological sort (BFS with in-degree tracking)<br><br><b>Steps:</b><ol><li>Build adjacency list; compute <code>indegree[course]</code></li><li>Queue all courses with <code>indegree == 0</code> (no prerequisites)</li><li>Process queue: pop course, append to <code>order</code>, decrement neighbors' in-degrees, enqueue any neighbor reaching <code>indegree == 0</code></li><li>Return <code>order</code> if <code>len(order) == numCourses</code>, else <code>[]</code></li></ol><b>Cycle detection is free:</b> nodes in a cycle never reach <code>indegree == 0</code> → never enter the queue → <code>len(order) &lt; numCourses</code>.<br><br><b>Complexity:</b> O(V + E) time and space<br><br><b>MLOps connection:</b> pipeline orchestrators (Airflow, Kubeflow) use this exact algorithm to schedule DAG steps and reject circular dependencies. Sliding Window Maximum — Return the max of each size-k window in O(n). What data structure do you use, what invariant does it maintain, and why store indices instead of values?	<b>Data structure:</b> Monotonic decreasing deque of <i>indices</i><br><br><b>Invariant:</b> values at stored indices are in decreasing order; <code>dq[0]</code> is always the index of the current window's maximum.<br><br><b>Per element i:</b><ol><li>Evict expired front: <code>while dq and dq[0] &lt; i - k + 1: popleft()</code></li><li>Maintain decreasing order: <code>while dq and nums[dq[-1]] &lt; nums[i]: pop()</code></li><li>Append <code>i</code></li><li>If <code>i &gt;= k - 1</code>: record <code>nums[dq[0]]</code></li></ol><b>Why store indices, not values?</b> You need to check whether <code>dq[0]</code> is still inside the current window — that requires the original position, not just the value.<br><br><b>Complexity:</b> O(n) time (each index pushed/popped at most once) | O(k) space	DSA MLOps
Find Median from Data Stream — Support addNum in O(log n) and findMedian in O(1). Describe the two-heap approach: structure, invariant, and the 3-step addNum procedure.	<b>Structure:</b><ul><li><code>lo</code>: max-heap (lower half) — store negated values in Python's <code>heapq</code></li><li><code>hi</code>: min-heap (upper half)</li></ul><b>Invariant after every addNum:</b> <code>len(lo) - len(hi)</code> is 0 or 1; every element in <code>lo</code> ≤ every element in <code>hi</code><br><br><b>addNum(num) — 3 steps:</b><ol><li>Push <code>num</code> to <code>lo</code></li><li>Move <code>lo</code>'s max to <code>hi</code> (enforces ordering invariant)</li><li>If <code>len(hi) &gt; len(lo)</code>: move <code>hi</code>'s min back to <code>lo</code> (enforces size invariant)</li></ol><b>findMedian:</b><ul><li>Odd count: return <code>-lo[0]</code></li><li>Even count: return <code>(-lo[0] + hi[0]) / 2.0</code></li></ul><b>Complexity:</b> O(log n) addNum, O(1) findMedian	DSA MLOps
Top K Frequent Elements — Return k most frequent elements in better than O(n log n). Give both the heap approach and the O(n) bucket sort approach. Why is bucket sort O(n)?	<b>Heap approach — O(n log k):</b><ol><li><code>count = Counter(nums)</code></li><li><code>heapq.nlargest(k, count, key=count.get)</code> — or maintain min-heap of size k manually</li></ol><b>Bucket sort approach — O(n):</b><ol><li>Count frequencies</li><li>Create <code>buckets[i]</code> = nums with frequency <code>i</code>; length = <code>len(nums) + 1</code></li><li>Scan from high to low frequency, collect until k elements gathered</li></ol><b>Why bucket sort is O(n):</b> frequency is bounded by <code>len(nums)</code> — at most n distinct frequency values — so counting sort applies to the frequency domain.<br><br><b>Interview move:</b> mention bucket sort as the optimal approach; implement heap if the bucket sort explanation is unclear. Number of Islands — Count connected land components in a binary grid. Describe BFS flood fill and explain why BFS is preferred over DFS in Python.	<b>Approach:</b> BFS flood fill (connected components on an implicit graph)<br><br><b>Algorithm:</b><ol><li>Scan every cell</li><li>When <code>grid[r][c] == '1'</code>: increment island count, start BFS from <code>(r, c)</code></li><li>BFS marks every reachable <code>'1'</code> as <code>'0'</code> to prevent revisits</li></ol><div style="font-family:monospace;background:#f5f5f5;padding:8px;border-radius:4px;margin:4px 0">queue = deque([(r, c)]); grid[r][c] = '0'<br>while queue:<br>&nbsp;&nbsp;row, col = queue.popleft()<br>&nbsp;&nbsp;for dr, dc in [(1,0),(-1,0),(0,1),(0,-1)]:<br>&nbsp;&nbsp;&nbsp;&nbsp;if in_bounds and grid[nr][nc] == '1':<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;grid[nr][nc] = '0'; queue.append((nr,nc))</div><b>Why BFS over DFS in Python?</b> DFS can hit Python's default recursion limit (~1000) on large grids. BFS uses an explicit queue with no stack depth concern.<br><br><b>Complexity:</b> O(m × n) time | O(min(m, n)) BFS space	DSA MLOps
Merge K Sorted Lists — Merge k sorted linked lists into one. Describe the min-heap approach. Why include list_index in the heap tuple?	<b>Algorithm:</b> Min-heap of size k<br><br><b>Steps:</b><ol><li>Push <code>(node.val, list_index, node)</code> for each list's head</li><li>Pop minimum triple; append <code>node</code> to result</li><li>If <code>node.next</code> exists: push <code>(node.next.val, list_index, node.next)</code></li><li>Repeat until heap empty</li></ol><b>Why <code>list_index</code> in the tuple?</b> Python can't compare <code>ListNode</code> objects. When two values are equal, Python tries to compare the next tuple element. <code>list_index</code> is an int and resolves ties without ever comparing nodes — otherwise you get a <code>TypeError</code>.<br><br><b>Complexity:</b> O(N log k) time where N = total nodes, k = lists | O(k) heap space<br><br><b>Alternative:</b> Divide and conquer — merge pairs recursively. Same O(N log k) time, O(log k) stack space.	DSA MLOps
Task Scheduler — Find minimum intervals to finish all tasks given cooldown n. What is the greedy insight? Give both the simulation and the O(1) math formula.	<b>Greedy insight:</b> Always execute the task with the highest remaining count. The most frequent task is the bottleneck — delaying it wastes the most future intervals.<br><br><b>Simulation — max-heap + cooldown queue:</b><ul><li>Max-heap: <code>(-count,)</code> per task type</li><li>Cooldown queue: <code>(remaining_count, available_at_time)</code></li><li>Each tick: re-enqueue expired tasks → pop heap → execute → push to cooldown</li></ul><b>O(1) math formula (follow-up):</b><br><code>result = max(len(tasks), (max_freq - 1) × (n + 1) + count_of_max_freq_tasks)</code><br><br><i>Logic:</i> (max_freq - 1) full frames of (n + 1) slots, plus one final slot per task tied at max frequency. If there are enough tasks to fill all idle slots, <code>len(tasks)</code> dominates.<br><br><b>Complexity:</b> O(n_tasks × n) simulation | O(26) = O(1) space	DSA MLOps
Word Break — Can string s be segmented into words from wordDict? Give the DP state, recurrence, base case, and the key implementation detail for efficiency.	<b>State:</b> <code>dp[i] = True</code> if <code>s[:i]</code> can be segmented into dictionary words<br><br><b>Recurrence:</b> <code>dp[i] = True</code> if ∃ <code>j &lt; i</code> such that <code>dp[j] == True</code> and <code>s[j:i] ∈ word_set</code><br><br><b>Base case:</b> <code>dp[0] = True</code> (empty string is trivially segmentable)<br><br><b>Answer:</b> <code>dp[len(s)]</code><br><br><b>Key implementation details:</b><ul><li>Convert <code>wordDict</code> to a <code>set</code> for O(1) substring lookup</li><li><code>break</code> after setting <code>dp[i] = True</code> — prunes the inner loop</li></ul><b>Complexity:</b> O(n²) time, O(n) space<br><br><b>Follow-up — Word Break II (return all sentences):</b> use memoized DFS, not DP — you need to reconstruct paths, not just True/False. Design Hit Counter — Count hits in the past 300 seconds. Give the deque approach AND the O(1)-space circular buffer follow-up. Explain the circular buffer's key invariant.	<b>Deque approach:</b><ul><li>Store <code>(timestamp, count)</code> pairs; merge same-timestamp hits</li><li><code>hit(t)</code>: if last entry same timestamp, increment; else append</li><li><code>getHits(t)</code>: evict <code>timestamp &lt;= t - 300</code> from front; return sum of counts</li><li>Complexity: O(1) amortized hit, O(n) getHits</li></ul><b>Circular buffer — O(1) space follow-up:</b><ul><li>Two arrays of size 300: <code>times[300]</code>, <code>hits[300]</code></li><li><code>hit(t)</code>: <code>idx = t % 300</code>; if <code>times[idx] == t</code>: increment; else reset both</li><li><code>getHits(t)</code>: sum <code>hits[i]</code> where <code>times[i] &gt; t - 300</code></li></ul><b>Key invariant:</b> <code>times[idx]</code> tells you which 300-second cycle this slot belongs to. Overwriting an old slot is safe — any entry &gt; 300 seconds ago is outside the window anyway.<br><br><b>Complexity:</b> O(1) hit, O(300) = O(1) getHits, O(1) space	DSA MLOps
