[K8s Architecture] Walk through everything that happens when you run <code>kubectl apply -f deployment.yaml</code> — from your terminal to running containers.	<ol><li><b>kubectl</b> reads YAML, sends PATCH/POST to the <b>API server</b> over HTTPS</li><li><b>API Server</b> authenticates, authorizes (RBAC), validates schema, runs admission webhooks (mutating → validating), <b>writes to etcd</b></li><li><b>Deployment controller</b> (in controller-manager) receives watch event → creates/updates a <b>ReplicaSet</b></li><li><b>ReplicaSet controller</b> creates <b>Pod objects</b> in etcd — unscheduled (no <code>nodeName</code>)</li><li><b>Scheduler</b> watches for unscheduled Pods → filters nodes (resources, taints, affinity) → scores → writes <code>nodeName</code> to Pod spec in etcd</li><li><b>kubelet</b> on target node receives watch event → calls <b>container runtime</b> (containerd) to pull image and start container</li><li>kubelet reports Pod status back; ReplicaSet controller sees desired state achieved</li></ol><b>Key principle:</b> every step is a control loop watching etcd. Kubernetes is eventually consistent — each component reconciles independently.	K8s Architecture
[K8s Architecture] etcd loses quorum. What still works in the cluster? What immediately breaks?	<b>Still works:</b><ul><li>Running Pods continue running — kubelet caches Pod specs locally; no etcd needed to keep containers alive</li><li>Existing kube-proxy iptables rules → Service routing continues</li><li>Application traffic flows normally — users see no disruption</li></ul><b>Breaks immediately:</b><ul><li>API server returns errors on writes; kubectl commands fail</li><li>Scheduler can't assign new Pods → new Pods stay Pending forever</li><li>Controllers can't reconcile → no self-healing (crashed Pod not replaced)</li><li>No config updates → no new Deployments, ConfigMaps, Secrets</li></ul><b>Key insight:</b> The control plane is the brain; the data plane (running workloads) keeps running without it — until something needs to change. [K8s Debugging] How do liveness, readiness, and startup probes differ? When is using liveness where readiness is correct a dangerous mistake?	<table border="1" cellpadding="4" style="border-collapse:collapse"><tr><th>Probe</th><th>Failure Action</th><th>When to Use</th></tr><tr><td><b>liveness</b></td><td>Kill + restart container</td><td>App can deadlock; only restart fixes it</td></tr><tr><td><b>readiness</b></td><td>Remove from Service endpoints (no restart)</td><td>App temporarily unable to serve (loading, upstream down)</td></tr><tr><td><b>startup</b></td><td>Kill + restart; disables liveness/readiness until passes</td><td>Slow-starting apps (JVM, large ML model load)</td></tr></table><b>The dangerous mistake:</b> A model-serving app becomes temporarily overloaded. A <i>liveness</i> probe fails → K8s restarts the Pod → Pod must reload the model (slow startup) → all pods restart simultaneously → thundering herd → cluster-wide outage.<br><br><b>Correct design:</b> Use <code>readinessProbe</code> — removes the overloaded pod from rotation temporarily. It recovers on its own. Liveness is only for apps that are permanently broken without a restart.<br><br><b>startupProbe trick:</b> <code>failureThreshold: 30, periodSeconds: 10</code> → allows 300s startup time without liveness killing the pod. [K8s Networking] How does CoreDNS resolve a Service name? What DNS formats are valid? What is ndots and why does it matter for performance?	<b>Every Pod's /etc/resolv.conf:</b><div style="font-family:monospace;background:#f5f5f5;padding:8px;border-radius:4px">nameserver 10.96.0.10   # CoreDNS ClusterIP<br>search default.svc.cluster.local svc.cluster.local cluster.local<br>ndots: 5</div><b>Valid formats:</b><ul><li>FQDN: <code>my-svc.my-ns.svc.cluster.local</code> → always works</li><li>Short (same namespace): <code>my-svc</code> → expanded by search path</li><li>Cross-namespace: <code>my-svc.other-ns</code> → works via search expansion</li></ul><b>ndots: 5 behavior:</b> if the name has fewer than 5 dots, K8s tries appending search domains first before treating it as absolute. <code>my-svc</code> (0 dots) → tries <code>my-svc.default.svc.cluster.local</code> first.<br><br><b>Performance concern:</b> Each lookup tries 3 search domains before the absolute name. This adds latency for external DNS names (like <code>s3.amazonaws.com</code>). Fix: use fully qualified names with a trailing dot for external calls, or set <code>ndots: 2</code> in Pod's dnsConfig. [K8s Storage] What does volumeBindingMode: WaitForFirstConsumer do? What problem does it solve?	<b>Default behavior (Immediate):</b> PV is provisioned as soon as the PVC is created. At that point, the scheduler has already picked a node (and therefore an AZ) → the PV is provisioned in the same AZ as the node → volume mounts successfully.<br><br><div style="font-family:monospace;background:#f5f5f5;padding:8px;border-radius:4px">apiVersion: storage.k8s.io/v1<br>kind: StorageClass<br>metadata:<br>&nbsp;&nbsp;name: fast-ssd<br>provisioner: ebs.csi.aws.com<br>volumeBindingMode: WaitForFirstConsumer  # ← fix AZ mismatch</div><b>When it matters most:</b> Multi-AZ clusters with topology-aware storage (EBS, GCE PD). Not needed for topology-agnostic storage (EFS, NFS).	K8s Storage

[K8s Architecture] What does kube-proxy do? How does traffic actually reach a Pod when you call a Service ClusterIP?	<b>kube-proxy</b> runs on every node as a DaemonSet.
